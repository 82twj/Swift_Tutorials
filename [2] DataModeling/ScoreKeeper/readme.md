<h1>ScoreKeeper</h1>

간단한 점수 기록 앱을 만들면서 `SwiftUI`의 상태 관리와 데이터 모델링 방식을 연습한 프로젝트입니다. 플레이어를 추가하고, 게임 규칙을 정한 뒤, 게임 진행 상태에 따라 점수를 기록하고 최종 우승자를 판별합니다.<br>


<h2>프로젝트를 통해 배운 핵심 내용</h2>

| 항목 | 내용 |
|---|---|
| 화면 상태를 enum으로 관리하는 방법 | `GameState`는 `.setup`, `.playing`, `.gameOver` 세 가지 상태를 가집니다. |
  
이렇게 상태를 `enum`으로 분리하면:  
  
- 현재 화면이 어떤 단계인지 명확해집니다.  
- 버튼 동작을 상태별로 다르게 처리할 수 있습니다.  
- 특정 UI를 활성화하거나 비활성화하는 조건을 깔끔하게 작성할 수 있습니다.  
  
예를 들어 이 프로젝트에서는:  
  
- 설정 화면은 `setup` 상태에서만 수정 가능  
- 게임 시작 후에는 점수 조정 가능  
- 게임 종료 후에는 우승자 표시 가능  
  
처럼 상태 중심으로 화면 동작을 제어하고 있습니다.

| 항목 | 내용 |
|---|---|
| 모델과 UI를 분리하는 데이터 모델링 | 점수판의 핵심 데이터는 `Scoreboard`와 `Player`로 분리되어 있습니다.  
  
- `Player`: 이름과 점수를 가지는 개별 참가자 모델  
- `Scoreboard`: 플레이어 목록, 게임 상태, 승리 조건을 관리하는 상위 모델  
  
이 구조를 통해 배운 점은, 화면 코드 안에서 모든 로직을 처리하기보다  
데이터와 규칙을 모델에 모아두면 재사용성과 테스트가 좋아진다는 점입니다.  
  
특히 `Scoreboard` 안에:  
  
- `players`  
- `state`  
- `doesHighestScoreWin`  
- `winners`  
- `resetScores(to:)`  
  
같은 속성과 기능이 모여 있어서, 앱의 핵심 규칙이 한 곳에 정리됩니다.

| 항목 | 내용 |
|---|---|
| 계산 프로퍼티로 결과를 도출하는 방식 | `winners`는 저장된 값이 아니라 계산 프로퍼티입니다.  
  
즉, 별도로 우승자를 저장하지 않고 현재 플레이어 점수와 게임 규칙을 바탕으로  
필요할 때마다 우승자를 계산합니다.  
  
이 방식의 장점은:  
  
- 중복 데이터 저장을 줄일 수 있음  
- 점수가 바뀌었을 때 별도 동기화 실수를 줄일 수 있음  
- 비즈니스 로직이 명확하게 드러남  
  
이 프로젝트에서는:  
  
- 최고 점수 승리면 가장 큰 점수 탐색  
- 최저 점수 승리면 가장 작은 점수 탐색  
- 같은 점수인 플레이어는 모두 우승자로 반환  
  
하도록 구현되어 있습니다.

| 항목 | 내용 |
|---|---|
| mutating 메서드로 값 타입 변경하기 | `Scoreboard`는 `struct`이기 때문에 내부 데이터를 바꾸는 함수는 `mutating`이 필요합니다.  
  
`resetScores(to:)` 메서드를 통해 모든 플레이어의 점수를 한 번에 초기화하는 부분에서  
값 타입을 수정하는 Swift 방식에 익숙해질 수 있었습니다.  
  
이 코드를 통해 배운 점:  
  
- `struct`는 기본적으로 값 타입  
- 내부 프로퍼티를 수정하는 메서드는 `mutating` 필요  
- 배열 안의 요소도 인덱스로 접근해 변경 가능

| 항목 | 내용 |
|---|---|
| @State와 @Binding을 활용한 데이터 흐름 | `ContentView`에서는 다음 상태를 직접 소유합니다.  
  
- `@State private var scoreboard`  
- `@State private var startingPoints`  
  
그리고 `SettingsView`에는 `@Binding`으로 값을 전달합니다.  
  
이 구조를 통해 배운 핵심은:  
  
- 상위 뷰가 상태를 소유  
- 하위 뷰는 바인딩을 통해 값을 읽고 수정  
- 데이터 흐름이 한 방향으로 유지됨  
  
즉, 설정 화면은 독립적인 화면처럼 보이지만 실제 데이터의 주인은 `ContentView`입니다.

| 항목 | 내용 |
|---|---|
| ForEach 바인딩으로 배열 요소 직접 수정하기 | `ForEach($scoreboard.players)` 형태를 사용해 각 플레이어를 바인딩으로 받고 있습니다.  
  
이 방식 덕분에 각 행에서:  
  
- `TextField`로 이름 수정  
- `Stepper`로 점수 수정  
  
을 직접 반영할 수 있습니다.  
  
이 부분은 SwiftUI에서 리스트나 배열 기반 UI를 다룰 때 매우 중요한 패턴입니다.

| 항목 | 내용 |
|---|---|
| 상태에 따라 UI를 바꾸는 선언형 사고 | 이 프로젝트는 "현재 상태가 무엇인가"에 따라 UI를 달리 보여주는 SwiftUI의 선언형 방식이 잘 드러납니다. |
  
예를 들어:  
  
- `setup` 상태에서는 `Start Game` 버튼 표시  
- `playing` 상태에서는 `End Game` 버튼 표시  
- `gameOver` 상태에서는 `Reset Game` 버튼 표시  
  
처럼 조건문으로 현재 상태에 맞는 화면을 선언합니다.  
  
이 과정에서 배운 점은,  
SwiftUI에서는 "어떻게 바꿀지"보다 "지금 어떤 상태이므로 무엇을 보여줄지"를  
기준으로 코드를 작성하는 것이 더 자연스럽다는 점입니다.

| 항목 | 내용 |
|---|---|
| 뷰 비활성화로 사용자 입력 제한하기 | `.disabled(...)`와 `.opacity(...)`를 함께 사용해 게임 단계에 따라 수정 가능한 영역과 불가능한 영역을 구분하고 있습니다.  
  
이를 통해:  
  
- 설정이 가능한 시점과 아닌 시점을 구분  
- 점수를 조절할 수 없는 상황을 시각적으로 표현  
- 사용자 실수를 줄이는 UI 구성  
  
을 배울 수 있었습니다.

| 항목 | 내용 |
|---|---|
| Swift Testing으로 핵심 로직 검증하기 | 테스트 코드에서는 다음 내용을 검증하고 있습니다.  
  
- 점수 초기화가 정상 동작하는지  
- 최고 점수 승리 규칙이 맞는지  
- 최저 점수 승리 규칙이 맞는지  
  
UI 테스트보다 먼저 모델 로직을 테스트한 점이 중요합니다.  
이 프로젝트를 통해 화면보다 규칙을 먼저 검증하는 테스트 습관의 중요성을 확인할 수 있습니다. |

<h2 style="color:#2563eb;">파일별 역할 정리</h2>

| 파일 | 역할 |
|---|---|
| `ScoreKeeperApp.swift` | 앱 시작 지점 |
| `ContentView.swift` | 메인 화면과 전체 흐름 제어 |
| `SettingsView.swift` | 게임 규칙과 시작 점수 설정 화면 |
| `GameState.swift` | 게임 진행 상태 정의 |
| `Player.swift` | 플레이어 모델 정의 |
| `Scoreboard.swift` | 점수판 데이터와 우승자 판정 로직 관리 |
| `ScoreKeeperTests.swift` | 점수판 로직 테스트 |

<h2 style="color:#2563eb;">이 프로젝트에서 특히 중요했던 포인트</h2>

이번 프로젝트의 핵심은 단순히 점수 앱을 만드는 것이 아니라, "상태를 기준으로 UI를 바꾸고, 규칙은 모델에 담고, 화면은 그 결과를 보여준다"는 SwiftUI의 기본 구조를 직접 연습한 데 있습니다.  
  
정리하면 다음 세 가지가 가장 중요했습니다.  
  
- `enum`으로 앱 상태를 명확하게 관리하기  
- `@State`와 `@Binding`으로 데이터 흐름 연결하기  
- 계산 프로퍼티와 메서드로 비즈니스 로직을 모델에 모으기 |

<h2 style="color:#2563eb;">개선해볼 수 있는 점</h2>

- 빈 이름 입력 방지  
- 플레이어 삭제 기능 추가  
- 동점자 처리 UI 개선  
- 승리 조건을 더 다양한 규칙으로 확장  
- 테스트에서 더 많은 경계 상황 검증 |

<h2 style="color:#2563eb;">한 줄 회고</h2>

이 프로젝트는 작은 앱이지만, SwiftUI에서 상태 관리, 값 타입 모델링, 바인딩, 계산 프로퍼티, 테스트까지  
실무적으로 자주 쓰이는 개념을 한 번에 연습할 수 있는 좋은 예제였습니다.
